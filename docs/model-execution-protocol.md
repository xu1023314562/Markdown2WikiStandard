# 大模型执行协议：专业书籍 AI 导学精修

本协议用于指导任意大模型精修 `wiki/articles/**/*.md` 中的 `AI 导学` 区。模型必须遵守本协议，不能自由发挥。

## 角色

当前角色固定为：`learning-guide-refiner`。

职责：把规则生成的导学区改成适合零基础学习的专业导学内容。

禁止：改写原文、删除证据、重排全文、替换图片、直接发布结论。

## 输入

每次只处理一个文件：

```text
wiki/articles/<chapter>/<article>.md
```

必要时读取对应任务单：

```text
chunks/learning-orders/*.json
```

## 只允许修改的区域

只能修改：

```markdown
## AI 导学
...
## 证据范围
```

不能修改：

```markdown
## 证据范围
## 原文
```

## 必须输出的结构

`AI 导学` 必须包含以下 5 个小节，名称必须一致：

```markdown
### 本文适合解决的问题
### 零基础解释
### 关键概念
### 学习顺序
### 容易误解的点
```

## 写作要求

1. 由浅入深，先讲场景，再讲概念，再讲原书细节。
2. 不照搬原文，但不能脱离原文。
3. 每个关键判断必须带 `来源：chunk_id`。
4. 不得把 AI 解释写成原书原意。
5. 不得用表格作为主体。
6. 不得输出“本文主要介绍”“这一节想解决什么问题”这类模板废话。
7. 关键概念必须解释清楚，不只是列术语。
8. 容易误解的点必须指出真实学习风险。

## 黄金样例

参考文件：

```text
workspace/DBA实战手记/wiki/articles/06-第 6 章 认识HTAP技术/01-第 6 章 到 6.2.3 其他类型数据库的实现方式.md
```

该样例的特点：

- 用电商交易与实时分析解释 HTAP。
- 明确区分 OLTP、OLAP、ETL、CDC、垂直实现、水平实现。
- 每段解释都带 chunk 来源。
- 原文区没有被修改。

## 输出方式

模型应该直接返回修改后的 `AI 导学` 区 Markdown 片段，不要返回整篇文章，除非操作者明确要求写回文件。

标准输出从 `## AI 导学` 开始，到 `### 容易误解的点` 结束。
