## 4. 遇到的问题与解决方案

### 4.1 🔴 JSONL 多行格式问题（关键踩坑）

**现象**：第一次写入的 `extracted/knowledge.batch-001.jsonl` 验证失败 213 个错误，全部是 "JSON 解析失败: Extra data"。

**根因**：用 `write` 工具直接写入多行 pretty-printed JSON，但 JSONL 规范要求**每行一个完整 JSON 对象**。

**解决**：
```python
# ❌ 错误写法（直接写入多行）
{"chunk_id": "x", "items": [...]}

# ✅ 正确写法（每行单 JSON 对象）
with open(path, 'w', encoding='utf-8') as f:
    for rec in records:
        f.write(json.dumps(rec, ensure_ascii=False) + '\n')
```

### 4.2 🔴 **流程偏差：精修任务走错目录**（本次新增踩坑）

**现象**：把"精修 markdown"任务理解为"新建概念页"，直接写入 `wiki/concepts/`。

**根因**：
- 没有重新读取 `model-execution-protocol.md` 第 3-12 行
- 把"精修"模糊等同于"自由生成 markdown"
- 忽略了 `model-execution-protocol.md` 已明确："精修 = 改 `AI 导学` 区"

**解决**：
- 把 6 个文件从 `wiki/concepts/` 移到 `review/candidates/`
- 在 frontmatter 标记 `status: draft`（待审计）
- 后续要走 `audit-concepts` 流程才能发布回 `wiki/concepts/`
- （更彻底的方案见 §6）

### 4.3 🟡 PowerShell GBK 编码崩溃

**现象**：在 PowerShell 控制台用 `print()` 输出含中文/特殊字符时崩溃。

**解决**：所有 Python 输出重定向到文件，不要 print 到控制台。

### 4.4 🟡 中文路径 read 失败

**现象**：`read` 工具对中文路径有时报 "File not found"，但 `bash Get-ChildItem` 能找到。

**解决**：先 `bash` 列出文件名 dump 到临时文件（UTF-8），再用 `read` 读临时文件。

## 7. 经验沉淀（关键教训）

### 7.1 流程层面

1. **路径决策前必须重新读取标准**：每次用户改向时，AI 必须先 `read model-execution-protocol.md` 确认路径，而不是凭印象决策。
2. **"精修"在 STANDARD 中有严格定义**：不是模糊词，对应"修改 `## AI 导学` 区"，不是"自由生成 markdown"。
3. **`wiki/concepts/` 是 publish 阶段产物**：必须先走 `review/candidates/` 审计，不能直接写入。
4. **task 类型决定路径**：
   - 抽取（P4）→ `extracted/`
   - 精修 AI 导学（P5）→ 改 `wiki/articles/`
   - 编译概念页（P6）→ `review/candidates/`
   - 发布（P8）→ `wiki/concepts/`

### 7.2 沟通层面

1. **用户说"精修"时应立即追问**：精修什么？AI 导学还是概念页？虽然可以直接给 1 个 question 即可。
2. **路径决策后主动告知替代选项**：让用户决策权始终在手上，而不是 AI 单方面决定。
3. **流程偏差要立刻承认并纠正**：不要为错误辩护，用户指出后立即按规范操作。

### 7.3 对 STANDARD 的改进建议

- **强制重新读取**：`STANDARD.md` 中"模型完成固定交接读取前"应扩展为"**每次任务类型变更时必须重新读取 model-execution-protocol.md**"。
- **任务分类清单**：在 `STANDARD.md` 中明确列出任务→路径映射表，避免 AI 凭印象决策。
