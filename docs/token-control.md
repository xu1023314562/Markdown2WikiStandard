# 专业书籍转换中的 Token 控制策略

你的现有问题是典型的“整书投喂 + 自由写作”失败模式：模型上下文被原书填满，输出却被压缩成低质量表格，既浪费 token，又没有证据质量。

## 禁止模式

不要这样做：

```text
读完整本书，帮我整理成 LLM Wiki。
```

原因：

- 输入 token 巨大，模型无法稳定保持全书结构。
- 输出目标不明确，模型倾向生成短摘要或表格。
- 没有证据 ID，后续无法审计。
- 每次追问都重新消耗大量上下文。

## 正确模式

```text
程序先读整本书，模型只读一个 chunk。
```

每个模型任务只包含：

- 一个 `chunk`。
- 一个严格提示词。
- 一个 JSON 输出 schema。
- 一个输出 token 上限。

## 预算建议

| 阶段 | 模型是否参与 | 输入规模 | 输出规模 |
|---|---|---:|---:|
| Markdown 解析 | 否 | 0 | 0 |
| 目录/分块 | 否 | 0 | 0 |
| 知识抽取 | 是 | 每块 800-1800 tokens | 每块不超过 900 tokens |
| 概念页编译 | 是 | 同一概念的证据集合 | 800-1500 tokens |
| Claim 审计 | 是 | 页面 + 证据 | 结构化 JSON |
| Obsidian 展示 | 否 | 0 | 0 |

## 省 token 的关键规则

1. 不让模型生成目录，目录由程序从 Markdown 标题提取。
2. 不让模型复制原文，原文已经在 `sources/`。
3. 不让模型写长文章，先抽取 JSON，再编译页面。
4. 不让模型反复读相同章节，所有 chunk 有 hash 和 ID。
5. 查询时读 `source-map.md` 定位 chunk，不回读全书。
6. 概念页只引用证据块，不把全章节塞给模型。

## 输出质量控制

低质量输出通常有三个特征：

- 只有表格，没有可追溯说明。
- 大量“本章主要讲了”这种空话。
- 没有 chunk 引用。

审计规则：

```text
如果输出不是 JSON 数组 → 拒绝
如果缺 source_chunk → 拒绝
如果只有表格 → 拒绝
如果出现无来源结论 → 进入 review
如果输出过短且 chunk 信息密集 → 重新抽取
```

## 推荐调用方式

每次只投喂一个任务单：

```text
读取 chunks/work-orders/sec-0123-c01.json。
严格执行其中 prompt。
只输出 JSON 数组，不要 Markdown，不要表格，不要总结全文。
```

如果模型有 5 小时限额，就不要在一次会话里处理整本书。按章节批处理，每批 10-20 个 chunk，处理完立即落盘。

## 批次处理

脚本会生成：

```text
chunks/batches.md
chunks/batches.json
```

建议每次只让模型处理一个 `batch`，并明确：

```text
按 batches.md 中 batch-001 的任务单逐个处理。
每个任务单独输出 JSON 数组。
把所有数组合并为 JSONL。
禁止输出总结文章、禁止输出 Markdown 表格。
```

这样可以把一次会话的输入输出控制在固定预算内，不会因为“读完整本书”耗尽 5 小时限额。
